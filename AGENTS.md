# AGENTS.md — astra-skill-execution-framework

> **For AI coding agents working on this project.**
> Read this first before making changes.

## Project North Star

A single-entry routing layer that tells the Hermes agent **what capabilities it needs** (not what the user wants) for any given task.

**Key shift (v2 → v3):** Stop classifying user intent ("deploy", "modify") — instead analyse the **agent's capability chain** needed to fulfil the task.

```
Example: "修一下服务器上的 SSH 配置"

Old way (user intent):     modify → load change-safeguard
New way (agent needs):     context → access → modification → closure
                           ↓            ↓         ↓              ↓
                      pre-action-  credential-  change-      work-
                      research     store       safeguard    closure-check
```

## Architecture (4 Layers)

| Layer | File | Purpose |
|:------|:-----|:--------|
| **Discovery** | `scripts/sync-routing.py --scan` | Scan skill directories, build index |
| **Index** | `skills_index.json` (generated) | Skill metadata: name, tags, indicators, description |
| **Recommend** | `scripts/recommend.py` | Match task description → ranked skill chain |
| **Trigger** | `SOUL.md §4` (temporary) | Agent self-calls recommend.py on task start |

### Dual Output Structure

Every recommendation returns two things:

1. **workflow** — Ordered steps appended to the agent's todo list.
   - `always: true` → mandatory for every task (context, closure)
   - `when: [...]` → semantic match triggers it (access, modification, deployment, push)

2. **background** — Behavioural rules injected into agent context for the whole task.
   - `mode: behavioral` → always active (work-principles)
   - `mode: active` → fires on specific phase (change-safeguard on modifying, vcs-discipline on executing)

## Key Files

| File | Role | Edit With Care? |
|:-----|:-----|:----------------|
| `routing.yaml` | **Single source of truth** for routing. Only file that maps task language → capability chains. | ✅ Yes — this is the primary edit target |
| `scripts/recommend.py` | Reads routing.yaml + skills_index.json, does keyword matching, prints recommendation | ⚠️ Low priority — logic is straightforward |
| `scripts/sync-routing.py` | Scans skill directories, generates skills_index.json | ⚠️ Low priority — rarely needs changes |
| `skills_index.json` | Auto-generated by `--scan`. Do not edit manually. | ❌ Never hand-edit |
| `SKILL.md` | Hermes agent instructions for when the skill is loaded | ⚠️ Only update if routing.yaml changes affect usage flow |
| `config.yaml` | Declares scan paths for --scan mode | ⚠️ Rarely changes |
| `SOUL.md` (in HERMES_HOME) | §4 contains the temporary self-call instruction | ⚠️ Replace with plugin hook when available |

## Design Decisions

### 1. routing.yaml as single source of truth

Not SKILL.md, not README.md. All routing logic lives in structured YAML so it's:
- Machine-readable (recommend.py parses it)
- Versionable (independent of prose docs)
- Editable (one file, clear schema)

### 2. Language-independent matching

Keywords in `when:` are bilingual (Chinese + English) heuristics, NOT an exhaustive list. The agent should use semantic understanding first, fall back to keywords.

### 3. workflow ≠ background

| | workflow | background |
|:--|:---------|:-----------|
| Checkable? | ✅ Yes — appended to todo, can be ticked off | ❌ No — always active |
| Ordered? | ✅ Sequential steps | ❌ Parallel rules |
| Skip? | Optional steps skipped if `when:` doesn't match | Never skip |
| Example | `pre-action-research → credential → change-safeguard → closure` | "诚实优先", "先保全再改" |

### 4. SOUL.md §4 is temporary

The auto-trigger currently lives as a patch in `~/.hermes/SOUL.md` (work principles §4). When a proper Hermes plugin hook (`on_task_start`) replaces it:
- Remove §4 from SOUL.md
- Update SKILL.md to remove "temporary" warnings
- Update AGENTS.md to reflect the new mechanism

### 5. Scope boundaries

| In scope | Out of scope |
|:---------|:-------------|
| Routing astra ecosystem skills | Hermes built-in skill selection |
| Mapping task language → capability chain | Providing per-skill execution checklists |
| Background discipline injection | MCP server routing |
| Self-evolution (detect new skills) | Auto-creating new skills |

## How to Contribute

### Adding a new capability to routing.yaml

```yaml
- id: my-capability
  label: "中文标签"
  order: N          # position in workflow
  when:
    - trigger1
    - 触发词2
  skills:
    - name: target-skill
```

1. Add the entry to `routing.yaml` under the correct `workflow` or `background` section
2. Run `python3 scripts/sync-routing.py --scan --write-index` to regenerate the index
3. Test with `python3 scripts/recommend.py "test phrase"` — verify it appears in the output

### Testing recommendations

```bash
# Test a specific task
python3 scripts/recommend.py "帮我把 Nginx 配置成反向代理"

# Test with structured output
python3 scripts/recommend.py --json "push this repo to GitHub"

# Interactive mode (type tasks repeatedly)
python3 scripts/recommend.py --interactive
```

### Before committing

- [ ] `routing.yaml` parses as valid YAML (`python3 -c "import yaml; yaml.safe_load(open('routing.yaml'))"`)
- [ ] `recommend.py "test phrase"` returns sensible output (not empty/crash)
- [ ] New keywords in `when:` have both Chinese and English variants where applicable
- [ ] `always: true` steps (context, closure) are never accidentally removed
- [ ] skills_index.json is regenerated if skills_index entries changed
